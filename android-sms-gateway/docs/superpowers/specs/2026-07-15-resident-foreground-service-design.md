# 普通 APK 常驻前台服务设计

## 背景

当前应用通过 `SSEForegroundService` 保持云端事件连接，并使用 WorkManager 周期拉取消息。该结构不能稳定承担整个应用的常驻职责：设备尚未注册、云网关关闭或 SSE 初始化失败时，应用没有独立的常驻载体；服务的 `onStartCommand()` 也没有明确返回 `START_STICKY`。

本次改造以普通 APK 为部署形态，不依赖 root、平台签名、系统应用权限或 Magisk。目标是在 Android 公共 API 允许的范围内提供最高可靠性的常驻运行。

## 目标与边界

### 目标

- 使用独立前台服务维持应用进程，并持续显示“短信基站运行中”通知。
- 系统回收进程后请求 Android 重建服务。
- 在开机完成和应用升级完成后恢复常驻服务。
- 保留常驻开关，默认开启，并让启停操作立即生效。
- 常驻服务重建后恢复 SSE、短信监听和 WorkManager 任务。
- 所有启动入口保持幂等，不重复注册监听器或创建重复任务。
- 单个业务模块启动失败时记录日志，不终止整个常驻服务。

### 明确边界

- 用户在系统设置中执行“强制停止”后，普通 APK 不能自行恢复；用户必须再次启动应用。
- 厂商 ROM 仍可能限制后台运行。应用继续提供忽略电池优化入口，但不能替用户修改厂商自启动白名单。
- 本次不使用高频 AlarmManager 唤醒、隐藏进程、双进程互拉或其他规避系统策略的机制。
- 本次不改变短信网关协议、消息队列语义或云端 API。

## 方案选择

### 方案一：仅强化 SSE 前台服务

改动最少，但 SSE 服务受云网关启用状态和设备注册状态约束，不能覆盖所有常驻场景，因此不采用。

### 方案二：独立常驻前台服务

新增只负责进程存活和业务编排的 `ResidentForegroundService`。SSE、Worker 和短信接收模块仍保持原有职责。该方案边界清晰，能在 SSE 暂时不可用时继续维持应用运行，因此采用此方案。

### 方案三：AlarmManager 或 WorkManager 高频自检

该方案耗电更高，并受新版 Android 后台启动限制，不适合作为主常驻机制，因此不采用。现有周期消息拉取仍作为业务兜底保留。

## 架构

### ResidentForegroundService

新增 `ResidentForegroundService`，作为应用常驻生命周期的唯一入口：

- 启动后立即调用 `startForeground()` 显示不可滑动清除的常驻通知。
- `onStartCommand()` 明确返回 `START_STICKY`。
- 调用 `OrchestratorService.start()` 恢复各业务模块。
- 捕获业务初始化异常并写入应用日志，避免异常导致前台服务退出。
- 服务被用户通过应用内开关关闭时，调用 `OrchestratorService.stop()`，移除通知并停止自身。
- 重复收到启动请求时只执行幂等的“确保运行”逻辑。

`ResidentForegroundService` 不承担 SSE 连接、短信处理或消息拉取逻辑，避免把业务职责耦合进进程守护组件。

### OrchestratorService

`OrchestratorService` 继续统一启动和停止业务模块，但需要具备幂等生命周期：

- 连续调用 `start()` 不重复注册动态广播接收器、ContentObserver 或事件监听器。
- 连续调用 `stop()` 不抛出未注册或重复停止异常。
- 一个模块启动失败不阻断其他模块启动。
- 常驻开关的判断集中在常驻服务启动入口，避免不同入口行为不一致。

### 恢复广播

保留 `BootReceiver`，并增加应用升级完成事件：

- 处理 `BOOT_COMPLETED` 和现有兼容性开机事件。
- 处理仅针对本应用的 `MY_PACKAGE_REPLACED`。
- 广播接收器只判断常驻开关并快速启动 `ResidentForegroundService`，不在 `onReceive()` 中执行完整业务初始化。
- 常驻开关关闭时，广播接收器不启动任何服务。

### WorkManager 兜底

保留 `PullMessagesWorker` 的即时任务和最短周期任务。它负责在 SSE 通知缺失时拉取待发消息，不承担常驻进程职责。

周期任务使用唯一任务策略维持单例。常驻服务重建时不得无条件重置已有周期任务的调度窗口。

## 设置与用户体验

- 保留现有“开机启动”开关，并将尚未保存过该配置时的默认值改为开启。
- 用户开启开关时，立即启动 `ResidentForegroundService`。
- 用户关闭开关时，立即停止常驻服务及全部业务模块和相关调度。
- 应用首次打开并获得运行所需权限后，按默认开启状态启动常驻服务。
- 常驻通知使用现有通知渠道和中英文资源体系，文案为“短信基站运行中”。
- 点击常驻通知打开 `MainActivity`。
- 应用继续提供忽略电池优化的设置入口。

## 生命周期与数据流

### 正常启动

1. 应用读取常驻开关。
2. 开关开启时启动 `ResidentForegroundService`。
3. 服务立即进入前台并显示通知。
4. 服务调用编排器确保短信监听、SSE 和各 Worker 正常运行。

### 系统回收后恢复

1. Android 根据 `START_STICKY` 重建前台服务。
2. 服务重新创建常驻通知。
3. 服务调用幂等编排器恢复业务模块。
4. 已存在的唯一周期任务保持原调度，不生成重复任务。

### 开机或应用升级后恢复

1. `BootReceiver` 收到开机或 `MY_PACKAGE_REPLACED` 广播。
2. 接收器读取常驻开关。
3. 开关开启时启动前台服务；关闭时直接返回。
4. 完整初始化由服务执行。

### 用户关闭常驻

1. 设置开关保存为关闭。
2. 应用停止编排器管理的业务模块和 WorkManager 任务。
3. 停止常驻前台服务并移除通知。
4. 后续开机和升级广播不再恢复服务。

## 错误处理

- 前台通知创建失败属于服务启动失败，保留系统异常并写入可用日志。
- 每个业务模块的启动异常分别捕获和记录，其他模块继续启动。
- 重复注册和重复注销必须通过内部运行状态或组件自身幂等保护消除。
- SSE 断线继续由现有重连机制处理，周期拉取作为兜底。
- 系统禁止后台启动前台服务时记录明确日志；应用下次由用户打开时再次恢复。

## 测试策略

### 自动化测试

- 新配置下常驻开关默认值为开启；已有用户保存的关闭值保持关闭。
- 开关开启触发常驻服务启动，关闭触发业务与常驻服务停止。
- 常驻服务的启动结果为 `START_STICKY`。
- 常驻服务重复启动不会重复初始化业务模块。
- 开机及 `MY_PACKAGE_REPLACED` 广播仅在开关开启时启动常驻服务。
- 编排器重复启停不会重复注册组件或抛出异常。
- 周期拉取任务重复确保运行时不重置已有任务。
- 运行现有 JVM 单元测试，防止短信和网关逻辑回归。

### 构建与手工验收

- 构建 Debug APK 成功。
- 首次安装并打开后，常驻开关默认开启且通知出现。
- 从最近任务中划掉应用后，通知仍存在或由系统粘性重启恢复。
- 锁屏待机、断网再联网后，SSE 与消息拉取恢复。
- 重启设备后，常驻通知和业务模块恢复。
- 覆盖安装升级后，常驻通知和业务模块恢复。
- 关闭开关后通知消失，重启设备也不恢复。
- 在系统设置中强制停止后应用不自行恢复，重新打开应用后恢复。

## 完成标准

当自动化测试和 Debug 构建通过，且手工验收覆盖任务划除、网络恢复、设备重启、覆盖升级和开关关闭场景时，本改造完成。强制停止后的不可自恢复属于普通 APK 的预期边界，不作为缺陷。
